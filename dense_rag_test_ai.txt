====================================================
一、测试环境与前置条件
====================================================

1. 所有测试在系统临时目录中展开（如 /tmp/dense_rag_test_XXXX），不能污染系统文件
   - 测试开始前创建临时目录作为 watch_dir
   - 在临时目录中生成专用的 config.yaml，将 watch_dir 指向该临时目录
   - 测试结束后（无论成功失败）清理临时目录和配置文件

2. 暂时只测试 .txt 文件的表现

3. embedding 服务前置检查
   - 测试脚本启动前，先检查 embedding 服务是否可达（如 curl 模型服务的 /v1/models 端点）
   - 若 embedding 服务不可用，应立即跳过全部测试并输出明确提示，而非让用例逐个超时失败

4. 服务启停管理
   - 测试脚本负责启动 dense-rag 服务（使用临时 config）
   - 等待服务就绪（轮询 /health 端点，最多等待 10s）
   - 执行全部测试用例
   - 测试结束后关闭服务进程、清理所有临时文件

5. score 判定策略
   - 由于采用向量检索存在误差，需根据返回的 score 判断用例是否通过
   - 建议先用一组已知文本做基准测试，确定 bge-m3 模型在当前场景下的 score 分布
   - 基于基准结果设定阈值（示例：score > 0.5 视为命中，score < 0.3 视为未命中）
   - 阈值应在脚本顶部定义为常量，便于调整

6. 等待与轮询策略
   - 文件写入后不要固定 sleep 2s 再查询，改为轮询方式
   - 每 500ms 查询一次，最多等待 10s，命中即通过，超时即失败
   - 这比固定 sleep 更稳健，不受 embedding 服务延迟波动影响
   - 原始需求中的"2s内可召回"作为参考指标，但测试容错应放宽


====================================================
二、核心功能测试（必测）
====================================================

TC-01 新增文件后可召回
  步骤：
    1. 在 watch_dir 下创建 test_add.txt，写入一段语义明确的文本
       （如"深度学习是人工智能的一个重要分支，通过多层神经网络进行特征提取"）
    2. 轮询 POST /query，查询语义相关文本（如"神经网络与深度学习"）
    3. 验证返回结果中包含该文件内容片段，且 score 超过命中阈值
  预期：文件被索引，语义相关查询可命中

TC-02 召回结果包含准确的文件路径
  步骤：
    1. 沿用 TC-01 已创建的文件
    2. 查询返回结果中的 file_path 字段
    3. 验证 file_path 为该文件在系统中的绝对路径
    4. 验证该路径确实存在于文件系统中（如 test -f "$file_path"）
  预期：file_path 为正确的绝对路径，且文件确实存在

TC-03 文件内容修改后可召回新内容
  步骤：
    1. 将 test_add.txt 的内容覆盖为全新文本
       （如"量子计算利用量子比特的叠加态和纠缠态来加速计算"）
    2. 轮询 POST /query，查询新内容相关文本（如"量子比特"）
    3. 验证可命中新内容，score 超过阈值
    4. 同时查询旧内容（如"神经网络与深度学习"），验证旧内容不再命中
       （score 低于未命中阈值，或结果中不包含该文件）
  预期：旧内容被替换，新内容可召回，旧内容不可召回

TC-04 文件删除后不可召回
  步骤：
    1. 删除 test_add.txt
    2. 等待一段时间后（轮询策略），查询 TC-03 中的新内容（"量子比特"）
    3. 验证结果中不包含该文件路径，或 score 低于未命中阈值
  预期：删除后该文件的所有向量条目被清除，无法召回


====================================================
三、/health 端点测试
====================================================

TC-05 健康检查基本功能
  步骤：
    1. 在 watch_dir 为空时请求 GET /health
    2. 验证返回 JSON 包含 status、vector_count、indexed_files 字段
    3. 验证 vector_count = 0，indexed_files = 0
  预期：返回格式正确，空目录时计数为零

TC-06 健康检查数值随文件增减变化
  步骤：
    1. 新增 health_test1.txt（写入一段文本）
    2. 等待索引完成（轮询 /health 直到 indexed_files > 0）
    3. 记录 vector_count 和 indexed_files
    4. 再新增 health_test2.txt
    5. 等待 indexed_files 增加
    6. 验证 indexed_files 比之前增加了 1
    7. 删除 health_test1.txt
    8. 等待 indexed_files 减少
    9. 验证 indexed_files 减少了 1，vector_count 也相应减少
  预期：/health 端点数值准确反映当前索引状态


====================================================
四、子目录递归监听测试
====================================================

TC-07 子目录中的文件可被索引和召回
  步骤：
    1. 在 watch_dir 下创建子目录 subdir_a
    2. 在 subdir_a 中创建 sub_test.txt，写入特定文本
       （如"自然语言处理使得计算机能够理解人类语言"）
    3. 轮询查询相关文本（如"计算机理解语言"）
    4. 验证命中，且 file_path 包含正确的子目录路径
  预期：递归监听生效，子目录文件可索引和召回

TC-08 动态新建子目录后再新增文件
  步骤：
    1. 在 watch_dir 下新建子目录 subdir_new（服务运行期间创建）
    2. 在 subdir_new 中创建文件 dynamic.txt
    3. 轮询查询该文件内容
    4. 验证可命中
  预期：watcher 动态注册新子目录，新目录下的文件变化可被捕获


====================================================
五、topk 数量限制测试
====================================================

TC-09 返回结果不超过 topk 条
  步骤：
    1. 创建 6 个以上 .txt 文件，每个文件内容语义相近但略有差异
       （如分别写入关于"机器学习"不同子主题的段落）
    2. 等待所有文件索引完成
    3. 查询一个泛化的相关关键词（如"机器学习算法"）
    4. 验证返回结果数量 <= topk（默认 5）
  预期：返回结果不超过配置的 topk 数量


====================================================
六、边界条件测试
====================================================

TC-10 空文件不崩溃
  步骤：
    1. 在 watch_dir 下创建一个空的 empty.txt（0 字节）
    2. 等待 2s
    3. 请求 /health 验证服务仍在运行
    4. 验证 /health 中 indexed_files 没有计入空文件（或计入但 vector_count 不增加）
  预期：服务不崩溃，空文件不产生向量条目

TC-11 大文件（>20MB）被忽略
  步骤：
    1. 在 watch_dir 下创建一个超过 20MB 的 big.txt
       （如用 dd if=/dev/zero bs=1M count=21 | tr '\0' 'A' > big.txt）
    2. 等待 2s
    3. 查询 big.txt 中的内容
    4. 验证不会命中该文件
    5. 请求 /health 验证 indexed_files 未增加
  预期：超过 20MB 的文件被忽略，不进行索引

TC-12 非支持格式文件被忽略
  步骤：
    1. 在 watch_dir 下创建 ignored.pdf、ignored.jpg、ignored.csv
       （内容可随意）
    2. 等待 2s
    3. 请求 /health 验证 indexed_files 未增加
  预期：非 .txt/.docx 格式的文件不被索引


====================================================
七、异常输入测试
====================================================

TC-13 空查询文本返回错误
  步骤：
    1. POST /query，body 为 {"text": ""}
    2. 验证返回 HTTP 400
  预期：空文本查询被拒绝

TC-14 无效 JSON 请求
  步骤：
    1. POST /query，body 为非法 JSON（如 "not json"）
    2. 验证返回 HTTP 400
  预期：格式错误被拒绝

TC-15 无索引时查询返回空数组
  步骤：
    1. 在 watch_dir 为空、无任何文件被索引时
    2. POST /query 查询任意文本
    3. 验证返回 HTTP 200，body 为空数组 []
  预期：无索引数据时不报错，返回空结果


====================================================
八、数据一致性测试
====================================================

TC-16 快速连续修改同一文件
  步骤：
    1. 创建 rapid.txt，写入初始内容 A
    2. 快速连续覆盖内容为 B、C、D（间隔 < 100ms）
    3. 等待索引稳定（轮询 /health 观察 vector_count 不再变化）
    4. 查询内容 D（最终内容）
    5. 验证可命中
    6. 查询内容 A/B/C（中间内容），验证不命中
  预期：debounce 机制生效，最终索引反映文件的最终状态

TC-17 文件改名（同目录内 rename）
  步骤：
    1. 创建 before_rename.txt，写入特定文本
    2. 等待索引完成
    3. 将文件重命名为 after_rename.txt（mv 命令）
    4. 等待索引更新
    5. 查询该文本内容
    6. 验证结果中 file_path 为新文件名 after_rename.txt
    7. 验证结果中不包含旧文件名 before_rename.txt
  预期：rename 触发旧路径删除 + 新路径索引，file_path 指向新名称


====================================================
九、用例执行汇总要求
====================================================

1. 每个用例独立可运行，互不依赖（TC-01 到 TC-04 可共享同一文件，但需标注执行顺序）
2. 每个用例输出 PASS / FAIL 及关键信息（返回的 score、file_path 等）
3. 最终输出汇总报告：总数、通过数、失败数、跳过数
4. 失败用例应输出完整的请求和响应，便于排查
5. 脚本退出码：全部通过返回 0，有失败返回非 0
