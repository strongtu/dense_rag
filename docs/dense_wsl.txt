WSL2 监控 Windows 文件夹 技术选型与优化清单
由master统筹确认可行性，再进行实施和合并

#背景：
由于要支持额外的监控目录，并且该目录在windows中
dense_rag可能运行在wsl环境中，无法通过现有技术方案实时监听如 /mnt/c/Users/strongtu之类目录的文件变化
因此需要在保持dense_rag功能稳定的前提下，进行优化
运行时判断处于WSL环境才运行该优化策略，以下是初步选型和技术要求

零、额外补充需求点
1. 原有config.yaml逻辑中，watch_dir要改为支持路径数组；同时支持旧配置格式单目录的解析
2. 需要对多目录做保护判断，避免用户配置有误，出现A是B的子目录的配置情况。如发生则退出并给出错误日志
3. 只有 'WSL环境'+'/mnt/xxx'形式的路径，才开启优化
4. 减少对原有监听机制pineline代码架构的侵入，如：
    a.保留原有Notify Watcher，新增轮询Watcher
    b.根据目录性质不同采用不同Watcher，避免交叉工作
    c.Watcher之后的处理工作，如清洗、向量计算应当是统一的流程
    d.首次启动时快速更新向量库，应当统一处理。随后才启动各自Watcher
5. 轮询Watcher针对统计到的变化要Debouncer处理

一、基础环境约束
 
1. 运行环境：WSL2
2. 监控目录：Windows 磁盘目录(/mnt/xxx)，不迁移至 WSL2 内部
3. 实现语言：Go，无第三方依赖，仅使用标准库
4. 无 Windows 侧辅助程序，纯 WSL2 内运行
 
二、文件量级与目标
 
1. 总文件数：1000~2000 个
2. 目标监控文件：数百个，按后缀过滤
3. 监控类型：文件新增、修改、删除
4. 监控范围：支持递归子目录
 
三、核心技术选型
 
1. 监听机制：基于轮询(因 WSL2 9P 协议不支持 inotify 事件透传)
2. 文件判断依据：文件 ModTime + 文件 Size 双重校验
3. 文件过滤：仅扫描指定后缀名文件，跳过无关文件与目录
 
四、性能优化参数
 
1. 轮询间隔：10000ms(10秒)，兼顾灵敏度与 CPU 占用
2. 遍历策略：仅遍历目标后缀文件，减少 I/O 与计算开销
3. 状态缓存：仅缓存上一轮目标文件状态，降低内存占用
 
五、支持的文件后缀(可扩展)
 
1. .txt
2. .docx
3. 自动忽略掉~$*.docx这类临时文件
 
六、监控事件类型
 
1. 文件新增
2. 文件修改(内容/大小/修改时间变化)
3. 文件删除
 
七、稳定性设计
 
1. 目录/文件读取错误捕获，不中断监听流程
2. 定时任务安全退出机制
3. 状态原子更新，避免比对异常
